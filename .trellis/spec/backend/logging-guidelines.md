# 日志与诊断记录规范

## 日志职责

普通日志用于运行时排障；诊断运行记录用于解释 AI 为什么得出某个结论。两者都重要，但不能互相替代。

## Python 日志

- Python Runtime 使用标准库 `logging`。
- 模块内使用 `logging.getLogger(__name__)`。
- 下游不可用、代理失败、发布/运维检查失败使用 `error` 或 `warning`。
- 调试信息不要污染 SSE 正常输出。

## 诊断运行记录

一次诊断应尽量记录：

- 运行基础信息：`run_id`、`session_id`、问题、状态、耗时、错误。
- 工具调用步骤。
- 上下文快照。
- 链路覆盖和应查项。
- 结构化证据。
- 根因摘要。
- 置信度和低质量原因。

AI 输出只是诊断运行的一个产物；运行步骤、覆盖率、证据和反馈同样是一等数据。

## 敏感信息

禁止记录：

- 密码、token、HMAC secret、生产凭据。
- 完整 Authorization header。
- 不必要的个人敏感信息。
- 未脱敏的生产连接串。

可以记录：

- 脱敏后的目标服务、状态码、耗时。
- 工具名称、步骤状态、错误类别。
- 目标 commit、image tag、发布 job id。

## 发布日志

- 长耗时容器构建日志保存在生产服务器 job 目录中，不应长期刷进 Codex 会话。
- Codex 启动异步发布后，应记录目标 commit / image tag，并在用户要求检查时运行 status、smoke 和 frontend hash。

## 常见错误

- 只记录最终回答，不记录证据链。
- 在聊天会话里持续刷远端构建日志。
- 把机密写入日志或文档。
- 诊断失败只有 traceback，没有用户可理解的失败原因。
