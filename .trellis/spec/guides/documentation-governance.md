# 文档治理规范

> **核心规则**：Trellis spec 是协作和开发规范源；`docs/` 是面向人类阅读的架构、运行、产品和历史材料入口。

## 文档归属

| 内容类型 | 目标位置 | 规则 |
| --- | --- | --- |
| 当前开发规范、AI 协作规范、代码约定 | `.trellis/spec/` | 任务执行前优先读取，必须保持可操作 |
| Trellis 流程、任务阶段、提交规则 | `.trellis/workflow.md` | 只放工作流规则，不放业务设计 |
| 单个任务的需求、设计、执行计划 | `.trellis/tasks/<task>/` | 跟随任务生命周期，完成后归档 |
| 当前架构、运行手册、部署说明、产品路线图 | `docs/` | 面向人类阅读，可引用 Trellis spec |
| 架构决策 | `docs/adr/` | 记录长期决策背景和取舍 |
| 阶段总结、评审报告、一次性审查报告 | `docs/archive/` | 历史参考，不作为当前事实源 |
| OMS 业务知识库 | `knowledge-base/` | RAG/业务知识正文，本规范不直接重写 |
| 根入口 | `README.md` / `AGENTS.md` / `CLAUDE.md` | 保持简短，只指向主要入口 |

## AI 协作规则分层

- `AGENTS.md`、`CLAUDE.md` 和 `README.md` 只放项目事实、主要入口和必须始终生效的少量约束；不要把完整编码规范塞进根入口。
- 具体“怎么写代码”的规则进入 `.trellis/spec/<layer>/`，并按 API、数据库、错误处理、日志、质量、发布等职责拆分。
- “写代码前要想什么”的检查清单进入 `.trellis/spec/guides/`，保持短 checklist，并指向对应 code-spec，不复制完整规则。
- 新增规则必须写成可执行表述：适用范围、正确/错误例子、验证方式和测试点要尽量明确，避免只写“注意规范”“保持一致”。
- 规则只在相关范围内加载或引用：改 API 读 API contract，改数据库读 database guidelines，改文档读本文档，避免无关规则挤占上下文和注意力。
- 强约束不能只依赖 AI 自觉遵守；能由 lint、类型检查、测试、脚本或 review checklist 验证的规则，必须配套验证手段。
- 规则需要持续演进：从 code review、线上问题和重复返工中提取规则；每个大版本或阶段收尾时清理过时、重复、冲突的条目。

## 新增文档规则

- 新增项目文档文件名中文优先，正文中文优先；英文仅用于专有名词、路径、命令、API、模型 ID、错误码、代码标识符和必要外部原文。
- 若必须保留英文文件名，必须有明确理由，例如生态约定、工具入口、外部规范或现有平台固定文件名。
- 新增“怎么做”的团队规范时，优先写入 `.trellis/spec/`。
- 新增“为什么这么设计”的长期决策时，优先写入 `docs/adr/` 或当前功能设计文档。
- 新增“这次任务做了什么”的过程记录时，写入 `.trellis/tasks/<task>/`，不要散落到 `docs/`。
- 创建 Trellis 任务时，任务标题、需求描述、验收标准、`prd.md`、`design.md`、`implement.md` 主体必须使用中文；不要用整句英文描述任务。
- 任务目录 slug 默认也使用中文。若当前工具链或脚本只支持 ASCII slug，可以使用拼音、中文首字母或短缩写作为路径兼容层，但 `task.json.title` 和任务正文仍必须是中文。
- `.trellis/tasks/<task>/` 下的 `prd.md`、`design.md`、`implement.md` 主体中文优先；路径、命令、字段名、组件名、类型名和必要专业名词可以保留英文。
- 新增阶段总结、评审报告、审查报告时，直接放入 `docs/archive/`，并在 `docs/README.md` 中登记。
- 如果 `docs/` 中的内容已经被迁移为 Trellis spec，原文只保留参考价值，不能再作为当前规范源。

## 维护检查

- [ ] 文档是否有唯一归属位置。
- [ ] 文件名和正文是否中文优先。
- [ ] 新建 Trellis 任务的标题、任务正文和规划材料是否使用中文。
- [ ] 英文是否仅用于专有名词、路径、命令、API、模型 ID、错误码、代码标识符或必要外部原文。
- [ ] 根入口是否保持简短，只指向事实源和关键入口。
- [ ] 如果是规范，是否已进入 `.trellis/spec/`。
- [ ] 规范是否按职责拆分，并只在相关任务范围内读取。
- [ ] 强约束是否有 lint、类型检查、测试、脚本或 review checklist 兜底。
- [ ] 是否清理了过时、重复或冲突的规则。
- [ ] 如果移动或归档文件，是否同步更新 `docs/README.md` 和仓库内引用。
- [ ] 是否避免把环境目录、缓存、构建产物写入 `docs/`。
- [ ] 是否保护 `knowledge-base/`，不和工程文档清理混在同一次迁移中重写。

## 错误做法

- 把长期开发规范只写在阶段总结里。
- 把大量可拆分规则塞进 `AGENTS.md`、`CLAUDE.md` 或 `README.md`。
- 只写抽象原则，不写适用范围、例子或验证方式。
- 把一次性评审报告放在 `docs/` 当前必读入口。
- 新建任务时用英文标题或英文 PRD 描述需求。
- 新增文档后不更新索引。
- 在 `docs/` 下创建 `.venv/`、缓存目录或构建产物。
